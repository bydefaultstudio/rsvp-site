What Is a Design System?

A Design System is your team's shared toolkit for creating consistent, high‑quality digital products and brand experiences.

Think of it as a well‑organised box of Lego bricks, but instead of plastic pieces, you get reusable components, brand assets, design rules, and clear guidelines.

It serves as a single source of truth for both product teams and brand guardians, so no one wastes time reinventing elements or guessing which visual style to use.

Whether you're designing an app screen, a marketing campaign, or a customer touchpoint, the Design System ensures everything looks, feels, and behaves like it comes from the same brand.

What's Inside a Design System?

  • Design tokens – Core values like colour palettes, spacing, typography scales, and animations that power both design and code.
  • Typography – Rules for fonts, sizes, and spacing to ensure readability and brand voice.
  • Components – Reusable interface elements like buttons, forms, and menus.
  • Brand assets – Logos, imagery, and iconography for consistent marketing and product use.
  • Layout principles – Guidelines for arranging content in a clear, intuitive way.

By following a Design System, teams can move faster, collaborate better, and maintain consistency across every brand interaction, whether it's in-app, on the web, or in the real world.

1. Design Tokens

Tokens are the foundation of a design system: they define the shared, reusable values that power both design and code, from colour and typography to spacing. By standardising these decisions in one place, tokens keep the experience consistent, reduce duplication, and make global updates safe and fast. Tokens describe values only, while classes apply those values with intent, so the system stays flexible without turning into one‑off styling.

Headings

All headings (h1–h4) use font-weight-bold (700).

  • H1 (font-3xl): Top-level page titles, hero sections, major headings. Uses line-height-s (1).
  • H2 (font-2xl): Section headings. Uses line-height-s (1).
  • H3 (font-xl): Sub-sections or supporting headings. Uses line-height-m (1.2).
  • H4 (font-l): Subheadings within content. Uses line-height-m (1.2).

Body Text

Body text uses font-m (18px desktop, 16px mobile) as the default size with line-height-l (1.4) for optimal readability.

  • font-m (18px): Default body size.
  • font-s (16px): Smaller body text.
  • font-xs (14px): Secondary text, captions, or supporting UI text.
  • font-2xs (12px): Metadata, labels, or disclaimers.

Custom / Utility Classes

  • font-3xl to font-2xs: Can be used as utilities for quick styling in UI contexts.

1.5 Spacing

Space tokens define distance, not intent.

They are reused for gaps, padding, and margins depending on context.

Space tokens are applied through layout and utility classes, not directly.

Global Padding

.padding-global applies left and right padding to section containers, ensuring consistent horizontal spacing across pages and components.

Centralising padding in this single token means updates at different breakpoints cascade automatically throughout the system — no need to adjust multiple component classes. Always apply .padding-global to main section containers. Use override classes (e.g. .full-bleed) only when a design explicitly requires breaking the horizontal grid.

Section Spacing

Section spacing controls the vertical rhythm between sections. It is handled through a combination of section spacing variables and utility combo classes. They scale responsively between desktop and mobile.

Variable Desktop Mobile
section-xs XL L
section-s 2XL XL
section-m 6XL 2XL
section-l 10XL 5XL
section-xl 14XL 8XL

These variables are not applied directly. They are consumed by the section utility classes described below.

.padding-section

.padding-section is the base class for all vertical section spacing.

On its own, it does not apply padding — it establishes a consistent pattern for spacing to be added via combo classes.

Always use .padding-section before applying any top or bottom spacing modifiers.

Top & bottom combo classes

Top and bottom spacing is controlled using combo classes, which map directly to the section spacing variables.

Size Top class Bottom class Uses variable
Small .top-small .bottom-small section-s
Medium .top-medium .bottom-medium section-m
Large .top-large .bottom-large section-l
XL .top-xl .bottom-xl section-xl

Each combo class applies its corresponding section-* variable as either padding-top or padding-bottom.

How it works (under the hood)

  • .top-large → applies padding-top using section-l
  • .bottom-large → applies padding-bottom using section-l
  • .top-small → applies padding-top using section-s
  • .bottom-small → applies padding-bottom using section-s

This keeps spacing values centralised, while allowing flexible composition at the layout level.

Padding Utilities

Padding utilities apply internal padding to elements. These are convenience utilities for quick spacing adjustments.

Class Spacing
.padding-s var(--space-s)
.padding-m var(--space-m)
.padding-l var(--space-l)
.padding-xl var(--space-xl)
.padding-2xl var(--space-2xl)
.padding-3xl var(--space-3xl)

Important

  • These utilities apply internal padding only (all sides).
  • They are convenience utilities for quick spacing adjustments.
  • They do not replace section spacing or layout primitives.
  • For section-level spacing, use section spacing combo classes.
  • For component-level spacing, prefer blocks with gap modifiers.

2. Border Strategy

Borders use a composable architecture that separates structural positioning from styling concerns. This approach keeps the system extensible without class explosion.

2.1 Core Principles

  • Structural classes define position only (.border, .border-top, .border-bottom, etc.)
  • Combo classes modify exactly one concern:
    • Width (.border-s, .border-m, .border-l)
    • Style (.border-solid, .border-dashed, .border-dotted)
    • Color (.border-primary, .border-secondary, .border-faded)
  • All border styling resolves through CSS variables
  • Do NOT create classes like .border-top-m or .border-bottom-l
  • Do NOT hardcode border values inside structural classes

2.2 Structural Classes

Structural classes define where the border appears, not how it looks.

Class Effect
.border Border on all sides
.border-top Border on top only
.border-bottom Border on bottom only
.border-left Border on left only
.border-right Border on right only

Example

<section class="border-bottom">
  <!-- Content with bottom border -->
</section>

2.3 Width Combo Classes

Width combo classes modify the border thickness.

Class Width Description
.border-s 1px Small border (default)
.border-m 2px Medium border
.border-l 4px Large border

Example

<div class="border border-m">
  <!-- Content with medium border on all sides -->
</div>

2.4 Style Combo Classes

Style combo classes modify the border appearance.

Class Style Description
.border-solid solid Solid border (default)
.border-dashed dashed Dashed border
.border-dotted dotted Dotted border

Example

<div class="border border-dashed">
  <!-- Content with dashed border -->
</div>

2.5 Color Combo Classes

Color combo classes modify the border colour using semantic tokens.

Class Color Description
.border-primary var(--border-primary) Strong border (default)
.border-secondary var(--border-secondary) Subtle border
.border-faded var(--border-faded) Faded border

Example

<div class="border border-secondary">
  <!-- Content with subtle border -->
</div>

2.6 Composing Borders

Combine structural classes with combo classes to create the exact border you need.

Example

<!-- Default border (small, solid, primary) -->
<div class="border-top">Content</div>

<!-- Medium dashed secondary border on bottom -->
<div class="border-bottom border-m border-dashed border-secondary">Content</div>

<!-- Large solid primary border on all sides -->
<div class="border border-l border-primary">Content</div>

2.7 Limitations

Native CSS dashed/dotted borders do not support adjustable dash spacing. This system intentionally avoids SVG or background-based borders to maintain simplicity and performance.

2.8 Rules

  • Use the smallest possible number of classes
  • Prefer semantic color tokens
  • Keep the system extensible without class explosion
  • Structural classes define position only
  • Combo classes modify one concern each
  • All styling resolves through CSS variables

3. Page Layout

All pages follow the same structural hierarchy:

3.1 Page Structure

body
└─ page-wrapper
  └─ page-content
    └─ section
      └─ padding-global
        └─ container / max-width
          └─ block

This structure ensures layouts are predictable, scalable, and easy to reason about.

  • body: Root of the document. Handles document-level semantics, accessibility landmarks, and global resets.
  • page-wrapper: Outermost layout wrapper that provides page-level context and handles overflow or global positioning rules.
  • page-content: Main content container that wraps all sections and separates page structure from global styles.
  • section: Represents a distinct content section. Groups related content and owns vertical spacing between sections.
  • padding-global: Applies consistent left and right padding to sections, maintaining horizontal rhythm across pages and breakpoints.
  • container / max-width: Constrains content width to maintain readable line lengths on larger screens.

3.2 Container & Max-Width Strategy

A container is a wrapper that constrains the width of content inside a section and centres it within the viewport.

It ensures:

  • readable line lengths
  • consistent layout width
  • predictable behaviour across breakpoints

Containers use max-width + width: 100% + auto margins, which naturally shrink on smaller viewports without requiring clamp().

Important

  • Containers do not include padding.
  • Padding should come from .padding-global or padding utilities.
  • Containers should always live inside padding-global.

Basic Usage

<section>
  <div class="padding-global">
    <div class="container-medium">
      <div class="block">
        <h2>Heading</h2>
        <p>Content</p>
      </div>
    </div>
  </div>
</section>

Variants

If multiple content widths are required, use explicit container variants.

Class Purpose
container-small Narrow layouts
container-medium Default readable width
container-large Wider layouts

Best Practice

  • One container per section (in most cases)
  • Containers always live inside padding-global
  • Do not nest containers unless explicitly required
  • Do not apply spacing, gaps, or margins to containers

If spacing is needed, use section spacing or blocks — not containers.

Max-Width

A container and a max-width do not do the same thing.

A max-width:

  • only applies a width constraint
  • does not centre content
  • inherits alignment from its parent

Because of this difference, containers are used as layout primitives, while max-width values are treated as an implementation detail.

Variants

Class Purpose
max-width-small Narrow width constraint
max-width-medium Default readable width
max-width-large Wider width constraint
max-width-full Full width

max-width-full

Use max-width-full when content needs to span the full available width of its parent, without introducing centring or container behaviour.

When to use

  • Edge-to-edge content within a section
  • Full-width media inside a constrained layout
  • Breaking out of a readable width while keeping the same layout structure

When not to use

  • As a replacement for containers
  • To centre content in the viewport
  • For standard page content

Rule of Thumb

  • If content should be centred in the viewport → use a container
  • If content only needs a width limit without centring → max-width may be used as an exception

By default, layouts should always rely on containers.

3.3 Block Strategy

A block is used to group related content and control spacing between the child elements. Use a block whenever content belongs together.

Common examples

  • Heading + body text
  • Text + buttons
  • Image + caption
  • Small content modules

If elements are related, they should live inside the same block.

Default Block Behaviour

By default, a block:

  • uses a vertical flex layout (display: flex; flex-direction: column)
  • uses gap to control spacing between children
  • applies a sensible default gap value
  • does not rely on margins

Child Sizing

Blocks are flex containers. Child sizing depends on the child's own display rules. Inline UI elements (e.g. .token-tag, .button) intentionally size to content and do not stretch.

Basic Usage

<div class="block">
  <h2>Heading</h2>
  <p>Supporting copy</p>
</div>

Gap Combo Classes

Use gap combo classes to change the spacing between elements inside a block.

Class Spacing size
gap-xs Extra small
gap-s Small
gap-m Medium (default)
gap-l Large
gap-xl Extra large

Example

<div class="block gap-l">
  <h2>Heading</h2>
  <p>More space between elements</p>
</div>

Layout & Alignment Combo Classes

Blocks can also accept layout and alignment modifiers.
These affect layout behaviour, not spacing values.

Class Effect
align-start Align items to start
align-center Centre-align items
align-end Align items to end
row Switch layout to horizontal
row-reverse Horizontal layout, reversed

Example

<div class="block row gap-m align-center">
  <p>Text</p>
  <button>CTA</button>
</div>

Rules

  • One block = one content group
  • Never add margins to elements inside a block
  • Use gap-* classes to control spacing
  • Use layout combo classes to control alignment or direction
  • Do not use spacer elements

If spacing feels wrong, adjust the block's gap or split content into multiple blocks.

3.4 Grid

Grids are layout primitives for creating column-based layouts. They are not spacing utilities.

Base Grid

.grid creates a two-column grid by default.

  • Display: grid
  • Gap: var(--space-m)
  • Columns: 1fr 1fr (equal width columns)

Column Modifiers

Class Columns Description
.cols-3 1fr 1fr 1fr Three equal-width columns
.cols-4 1fr 1fr 1fr 1fr Four equal-width columns
.token-col 1fr auto First column fills, second fits content

Grid Item Width Modifiers

Class Effect
.fit-content Item sizes to content width instead of filling column

Usage

<!-- Two-column grid -->
<div class="grid">
  <div>Item 1</div>
  <div>Item 2</div>
</div>

<!-- Three-column grid -->
<div class="grid cols-3">
  <div>Item 1</div>
  <div>Item 2</div>
  <div>Item 3</div>
</div>

<!-- Mixed sizing: one fills, one fits -->
<div class="grid token-col">
  <div>Fills remaining space</div>
  <div class="fit-content">Fits content</div>
</div>

Rules

  • Grids are layout primitives, not spacing utilities.
  • Use gap modifiers (.gap-*) to control spacing between grid items.
  • Combine column modifiers with gap modifiers as needed.

4. Typography & Text Styles

4.1 Links

Links use semantic color tokens and include hover states for better interactivity.

Default Link Styles

  • Color: var(--text-plain) (black)
  • Decoration: Underline with var(--text-link) color
  • Underline offset: 2.5px
  • Underline thickness: 1.5px
  • Transition: 0.3s for all properties

Hover State

  • Color: var(--text-link) (brand-1)
  • Underline offset: 4px (increased from 2.5px)

Links automatically use the design system's color tokens and provide smooth transitions on hover.

4.2 Eyebrow Text

The .eyebrow class is used for small, uppercase labels that appear above headings or sections.

Styles

  • Font family: Primary font
  • Font size: var(--font-s) (16px)
  • Line height: var(--line-height-m) (1.2)
  • Font weight: var(--font-weight-extra-bold) (800)
  • Letter spacing: 1.82px
  • Text transform: Uppercase

Usage

<h2 class="eyebrow">Section Label</h2>
<h1>Main Heading</h1>

4.3 Token Labels

Token labels are used to display design system token names in documentation and style guides.

.token-tag

A simple block-level label for displaying token names.

  • Display: block
  • Font size: var(--font-2xs) (12px)
  • Font family: Tertiary font (monospace)
  • Color: var(--text-faded)

.token-tag

A styled inline-block tag for displaying token names with background. Sizes to content and does not stretch.

  • Display: inline-block
  • Font size: var(--font-2xs) (12px)
  • Font family: Tertiary font (monospace)
  • Color: var(--text-plain)
  • Background: var(--background-faded)
  • Padding: var(--2xs) top/bottom, var(--s) left/right

Usage

<span class="token-tag">h2</span>
<h2>Heading</h2>

Rule

Labels, tags, badges, and UI chips should always size to content.

5. Buttons

Buttons are interactive elements that size to content by default.

5.1 Base Button

.button (or button element) provides the base button styling.

Styles

  • Font: Primary font, var(--button-font-size) (14px)
  • Font weight: var(--font-weight-bold) (700)
  • Text transform: Uppercase
  • Display: flex (inline-block fallback)
  • Padding: var(--space-l) vertical, var(--space-2xl) horizontal
  • Background: var(--button-primary)
  • Color: var(--button-text)
  • Border: var(--border-m) solid var(--button-primary)

Important

  • Buttons size to content and do not stretch full width by default.
  • Buttons use flexbox for internal alignment.

5.2 Button Modifiers

Class Effect
.is-outline Transparent background, colored border and text
.is-faded Faded background using var(--button-faded)
.is-small Smaller font size and padding
.is-icon Circular button with equal padding

Usage

<button class="button">Default Button</button>
<button class="button is-outline">Outline Button</button>
<button class="button is-faded">Faded Button</button>
<button class="button is-small">Small Button</button>
<button class="button is-icon">
  <div class="icn-svg"><!-- SVG icon --></div>
</button>

Modifier Combinations

  • .is-outline.is-faded - Outline button with faded border color
